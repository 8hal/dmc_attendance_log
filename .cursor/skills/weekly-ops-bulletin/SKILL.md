# 주간 운영진 안내 (Weekly ops bulletin)

## 언제 쓰는가

- **주간 회고가 끝난 뒤**, 카톡·이메일·노션 등으로 **운영진에게 한 주 요약**을 보낼 때.
- 수신자는 **IT 배경이 거의 없음**. 짧은 문장, 불릿, 전문 용어 금지.

## 절차

1. **근거 수집:** `_docs/log/YYYY-MM-DD.md` 해당 주 분(월~일)과 필요 시 최근 커밋 메시지를 읽는다. 대화만의 기억으로 쓰지 않는다.
2. **사실 검증:** 코드·일지에 없는 성과는 단정하지 않는다. 불확실하면 "진행했다" 수준으로 쓴다.
3. **기술 표현 교정:** 함수명·API명·컬렉션명·배포 URL을 운영진 문구에 절대 노출하지 않는다.
4. **구성 원칙 — 임팩트 중심, 짧게:**
   - **지난 주:** 운영진이 알면 좋은 일 **2~3불릿만**. 임팩트 있는 것만 고른다.
     - 좋은 예: "데이터·시스템 점검 목표 90% 완료"
     - 나쁜 예: "searchOhmyrace() 함수 파싱 로직을 .name-card 기반으로 수정"
   - **이번 주:** 뭘 할지 **1~2불릿**. 운영진 관점의 결과물로 표현.
     - 좋은 예: "단체 대회 참가자를 미리 입력해두면 대회 후 기록이 자동으로 수집·저장되는 기능 개발 시작"
     - 나쁜 예: "race_events 컬렉션 + participants 배열 + Cloud Scheduler 자동 트리거 구현"
   - **부탁:** 넣지 않는다. 필요시 사용자가 직접 추가.
5. **채널 형식:** `■` 섹션 + `·` 불릿, 블록 사이 빈 줄.
6. **길이:** 본문 **8줄 이내**. 짧을수록 좋다.
7. **수치:** 일지·API 출력에 없으면 쓰지 않는다.

## 정식 예시 (2026-04-06 사용자 승인본)

```
동마클 기록 시스템 주간 공유

■ 지난 주에 한 일

· 데이터·시스템 점검 목표 90% 완료

· 군산 새만금 마라톤 대응을 위해 새로운 기록 사이트 조회 기능 추가

· 일부 미완성 부분이 있어 추가 수정 진행 중

■ 이번 주에 할 일

· 단체 대회 참가자를 미리 입력해두면 대회 후 기록이 자동으로 수집·저장되는 기능 개발 시작
```

**핵심 기준:**
- 운영진이 읽고 1초 안에 이해할 수 있어야 한다.
- 기술 용어, 함수명, API명 금지.
- 불릿 하나가 한 문장을 넘으면 안 된다.

## 관련 룰·스킬

- `.cursor/rules/weekly-ops-bulletin.mdc` — 트리거·금지 패턴 요약
- `end-of-day` / 회고 로그 — 입력 소스가 `_docs/log`인지 확인

## 금지

- 배포 URL, 함수 이름, 컬렉션명을 운영진 문구에 노출하지 않는다.
- 검증 없는 구체 숫자 단정.
- 운영진에게 "직접 QA해 달라"고 쓰는데, 실제로는 담당자만 QA하는 경우.
