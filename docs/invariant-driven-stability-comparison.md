# 불변식 중심 안정성 설계 비교

## TL;DR

| 관점 | 무엇을 보나 | 주로 생기는 착각 | 필요한 작업이 늘어나는 이유 |
|---|---|---|---|
| A. 기능 중심 개발 | 화면 1개, 버튼 1개, 요청 1개 | "여기만 고치면 된다" | 시스템 전체의 연결 규칙을 놓치기 쉽다 |
| B. 불변식 중심 설계 | 데이터 흐름, 식별자, 상태 전이, 권한 경계 | "작은 수정도 시스템 계약을 건드릴 수 있다" | 테스트, 검증, 분리, 로그, privacy 작업이 자연스럽게 따라온다 |

핵심은 이렇다.

- **기능은 로컬에서 보이지만, 장애는 경계(boundary)에서 발생한다.**
- 그래서 안정성 작업은 각자 따로 필요한 것이 아니라, **"시스템 경계를 안전하게 다루기 위한 공통 원리"**에서 나온다.
- 이 프로젝트에서 그 공통 원리는 크게 5가지다.
  - **불변식(invariant)**
  - **SSOT(Single Source of Truth)**
  - **상태 전이(state transition)**
  - **멱등성(idempotency)**
  - **최소 권한/최소 노출(least privilege / least exposure)**

---

## 배경

이 프로젝트는 단순한 정적 페이지가 아니다.

- Firestore 컬렉션 여러 개가 연결되어 있고
- 외부 기록 사이트를 조회하고
- 공개 화면과 운영 화면이 섞여 있으며
- 검색 결과를 다시 확정 데이터로 승격시키는 흐름이 있다.

즉, 이 시스템은 **"입력 → 임시 상태 → 확정 상태 → 공개 소비"**가 있는 작은 분산 시스템에 가깝다.

이런 시스템에서 사고가 나는 이유는 보통 코드가 복잡해서가 아니라,

> **서로 다른 부분이 암묵적으로 공유하던 약속이 깨졌는데도, 그 약속이 코드에서 강제되지 않았기 때문**

이다.

대표 사례가 이번 기록 소실 이슈였다.

`race_results.jobId`와 `scrape_jobs` 문서 ID가 같은 값이어야 한다는 약속이 있었는데, 저장 시 raw search job ID가 들어가면서 조회에서 연결이 끊겼다.

```1078:1145:functions/index.js
if (action === "confirm" && req.method === "POST") {
  const { jobId, eventName, eventDate, source, sourceId, results } = req.body || {};

  const canonicalJobId = (source && sourceId && source !== "manual")
    ? `${source}_${sourceId}`
    : jobId;

  for (const r of results) {
    // ...
    batch.set(ref, {
      jobId: canonicalJobId,
      eventName: eventName || "",
      // ...
    });
  }
}
```

이 코드는 단순 저장 코드가 아니라, 사실상 **시스템 전체의 참조 무결성**을 지키는 코드다.

## A 상세 설명: 기능 중심 개발

### 비유

기능 중심 개발은 집의 한 방만 보고 인테리어를 바꾸는 방식과 비슷하다.

- 벽지를 바꾸고
- 전등을 더 예쁘게 달고
- 문 손잡이를 바꾼다

겉으로는 깔끔해 보인다.
하지만 그 문이 **방화문인지**, 전등 배선이 **다른 회로와 연결되어 있는지**, 벽이 **내력벽인지**를 모르고 손대면 나중에 큰 문제가 난다.

### 이 관점의 특징

- 변화의 단위가 UI 컴포넌트나 API 한 개다
- "지금 보이는 문제"를 빨리 고친다
- 국소적으로는 생산성이 높다

하지만 이런 질문은 놓치기 쉽다.

- 이 필드는 어디에서 읽히는가?
- 같은 값을 다른 이름으로 쓰는 곳이 있는가?
- 공개 화면이 운영 데이터까지 같이 받는가?
- 저장을 두 번 하면 어떤가?
- 실패했을 때 재시도하면 안전한가?

### 코드로 보면

`members` API는 "회원 목록을 내려주는 단순 API"처럼 보인다.
하지만 공개 페이지에서 바로 사용되면, 이건 사실상 **개인정보 공개 API**가 된다.

```817:825:functions/index.js
if (action === "members") {
  const snap = await db.collection("members").where("hidden", "==", false).get();
  const members = [];
  snap.forEach((doc) => {
    const d = doc.data();
    members.push({ id: doc.id, realName: d.realName, nickname: d.nickname, gender: d.gender || "", team: d.team || "" });
  });
  return res.json({ ok: true, members });
}
```

기능 중심 개발에서는 이 코드를 보며 이렇게 생각하기 쉽다.

- "회원 선택 UI에 필요한 데이터니까 내려주자"

하지만 시스템 관점에서는 질문이 달라진다.

- "정말 공개 클라이언트가 `realName`까지 알아야 하나?"

### 왜 이 관점만으로는 불안정해지는가

기능 중심 사고는 **행복 경로(happy path)** 에 강하다.
문제는 실제 서비스는 행복 경로만으로 운영되지 않는다는 점이다.

- 중복 저장
- 재시도
- 부분 실패
- 데이터 마이그레이션
- 외부 시스템 지연
- 공개/비공개 데이터 경계

이런 상황에서는 "버튼이 눌린다"보다 "시스템 약속이 유지된다"가 더 중요하다.

## B 상세 설명: 불변식 중심 설계

### 비유

불변식 중심 설계는 철도 시스템을 설계하는 것과 비슷하다.

기차 한 대가 잘 달리는 것보다 중요한 건,

- 신호 체계가 맞는지
- 선로 번호가 일치하는지
- 잘못된 선로로 들어갈 수 없는지
- 충돌이 났을 때 원인을 추적할 수 있는지

이다.

즉, **개별 동작의 성공보다 전체 시스템의 안전한 일관성**을 먼저 본다.

### 핵심 개념 1: 불변식

불변식은 "항상 참이어야 하는 약속"이다.

이 프로젝트에서 대표적인 불변식은 이미 문서화되어 있다.

```323:339:docs/DATA_MODEL.md
scrape_jobs doc ID = canonicalJobId = `${source}_${sourceId}` (non-manual)
race_results.jobId = 반드시 canonicalJobId와 동일

confirmed-races API:
  race_results.where("jobId", "==", scrapeJobDoc.id)

위반 사례: my.html 검색 저장 시 raw jobId(`search_xxx`)를 race_results에 저장
→ confirmed-races에서 canonical ID로 조회 시 매칭 안 됨 → 기록 "소실"
```

여기서 배우는 이론은 단순하다.

> **시스템이 여러 저장소/화면/API에 걸쳐 있으면, 연결 규칙이 곧 서비스의 진실이 된다.**

그래서 안정성을 높이려면 "기능 추가"보다 먼저 "절대 깨지면 안 되는 약속"을 명시하고 강제해야 한다.

### 핵심 개념 2: SSOT

SSOT는 "확정된 진실이 어디에 있는가"를 고정하는 원리다.

이 프로젝트에서는 `race_results`가 SSOT다.
왜냐하면 `scrape_jobs.results`는 검토용 원본이고, 확정 후에는 `race_results`만 믿어야 하기 때문이다.

SSOT가 필요하다는 건, 곧 이런 작업들이 필요하다는 뜻이다.

- 조회 경로 통일
- 확정 전/후 상태 분리
- 원본과 확정본의 책임 분리

즉, **데이터가 여러 군데 있더라도 "정답은 한 군데"** 여야 한다.

### 핵심 개념 3: 상태 전이

`running -> complete -> confirmed`는 단순 문자열이 아니라, 시스템이 밟아야 하는 생명주기다.

상태 전이를 명확히 다루지 않으면 이런 문제가 생긴다.

- complete인데 조회는 confirmed처럼 동작
- failed인데 UI는 계속 polling
- confirm 도중 일부만 저장

그래서 안정성 작업 중 "상태 전이 규칙 정의", "실패 시 복구", "부분 완료 차단"이 필요한 것이다.

### 핵심 개념 4: 멱등성

멱등성은 같은 요청을 여러 번 보내도 결과가 더 나빠지지 않는 성질이다.

서비스에서는 사용자가 두 번 클릭하지 않아도 멱등성이 필요하다.

- 네트워크 재시도
- 브라우저 새로고침
- 백엔드 재호출
- 배치 작업 재실행

같은 저장을 두 번 해도 결과가 안전해야 하므로,

- 결정적 Doc ID
- 중복 방지 키
- overwrite 규칙
- confirm 회귀 테스트

가 필요해진다.

즉, 멱등성은 "중복 클릭 방지"가 아니라 **실서비스의 재실행 안전성** 문제다.

### 핵심 개념 5: 최소 권한 / 최소 노출

공개 페이지에 실명이 내려가는 문제는 단순 문구 문제가 아니다.

```7:23:docs/PRIVACY_HARDENING_TODO.md
- `race?action=members` 응답에 회원 `realName`이 포함된다.
- `race?action=confirmed-races` 응답에 기록별 `realName`이 포함된다.
- `my.html`, `races.html`는 공개 페이지인데도 내부적으로 `realName`을 키로 사용한다.
- 일부 외부 기록 링크는 이름을 query string에 포함한다.
```

여기서의 이론은 보안 기본 원칙과 같다.

> **어떤 구성요소도 자신의 역할에 필요한 최소한의 정보만 가져야 한다.**

공개 클라이언트가 닉네임만 있으면 되는데 실명까지 받는다면,
그건 아직 기능은 돌아가도 설계는 불안정한 상태다.

그래서 privacy 강화 작업은 별도 분야가 아니라, 사실은 **안정성 설계의 일부**다.

## 핵심 차이 (Decision Tree)

```text
변경하려는 대상이 단순 UI 문구인가?
├─ 예 → 로컬 수정으로 끝날 가능성이 높다
└─ 아니오
   ├─ 저장소 키 / 식별자 / 참조 관계를 건드리나?
   │  └─ 예 → 불변식 검토 + 회귀 테스트 필요
   ├─ 확정 전/후 데이터 흐름을 건드리나?
   │  └─ 예 → SSOT / 상태 전이 검토 필요
   ├─ 같은 요청이 재실행될 가능성이 있나?
   │  └─ 예 → 멱등성 설계 필요
   ├─ 공개 화면에서 개인정보/운영 데이터를 받나?
   │  └─ 예 → API 분리 + 최소 노출 설계 필요
   └─ 외부 시스템 호출 실패가 서비스에 영향을 주나?
      └─ 예 → timeout / retry / observability 필요
```

이 트리는 곧 "안정성 작업이 왜 늘어나는가"에 대한 답이다.

작업이 많아 보이는 이유는 과잉 대응이 아니라,
**변경이 시스템 경계를 건드리는 순간, 검토해야 할 축이 늘어나기 때문**이다.

## 내 케이스 vs 상대 케이스 비교

### 내 케이스: 동호회 기록 시스템

- 공개 화면과 운영 화면이 함께 존재
- 회원 개인정보(실명)가 검색 매칭에 필요
- 외부 기록 사이트와 연동
- 임시 검색 결과가 확정 데이터로 승격
- 한 번 꼬이면 회원 입장에서 "기록이 사라졌다"로 체감

즉, 이 프로젝트는 **CRUD 앱보다 훨씬 신뢰성 요구가 높은 편**이다.

### 상대 케이스 1: 단순 소개 페이지

- 서버 상태 없음
- 데이터 저장 없음
- 외부 연동 없음

이 경우엔 "기능 중심 개발"만으로도 충분한 경우가 많다.

### 상대 케이스 2: 내부 관리자 전용 툴

- 사용자 수 적음
- 데이터 노출 범위 좁음
- 운영자 교육 가능

이 경우는 privacy와 UX 기준이 공개 서비스보다 느슨할 수 있다.

### 왜 내 케이스는 더 엄격해야 하나

이 프로젝트는 규모는 작아도 성격은 다음에 가깝다.

- **공개 서비스**
- **운영 시스템**
- **개인정보 처리**
- **외부 의존성 있는 데이터 파이프라인**

그래서 안정성 작업은 "나중에 하면 좋은 것"이 아니라,
서비스가 커지기 전에 시스템을 깨지지 않게 만드는 **기초 공사**다.

## 교훈

안정성을 위해 해야 하는 작업이 많은 이유는, 작업이 제각각 중요해서가 아니다.

모든 작업은 결국 같은 질문에 답하기 위해 존재한다.

> **이 시스템이 경계를 넘을 때도 같은 진실을 유지하는가?**

그래서 앞으로는 개별 작업을 외우기보다, 아래 순서로 생각하면 된다.

1. **이 변경이 어떤 경계를 건드리는가?**
   - UI / API / DB / 외부 사이트 / 공개-비공개 경계
2. **여기서 절대 깨지면 안 되는 약속은 무엇인가?**
   - 식별자, 참조, 상태, 권한
3. **그 약속을 코드가 강제하고 있는가?**
   - 검증, 테스트, 결정적 키, 상태 규칙
4. **실패했을 때 원인을 알 수 있는가?**
   - 로그, 모니터링, 복구 절차
5. **재실행해도 안전한가?**
   - 멱등성, 중복 방지, rollback-safe 설계

한 문장으로 줄이면 이렇다.

> **안정성 작업은 기능 개발의 반대가 아니라, 기능이 시스템 전체에서 계속 참이 되도록 만드는 비용이다.**
