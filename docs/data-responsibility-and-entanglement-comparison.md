# 데이터 책임 분리와 꼬임 비교

## TL;DR

| 관점 | 데이터가 어떻게 보이나 | 흔한 오해 | 실제로 필요한 사고방식 |
|---|---|---|---|
| A. 한 덩어리 데이터 관점 | "다 기록 관련 데이터잖아" | 비슷한 곳에 저장해도 된다 | 데이터마다 책임과 신뢰도가 다르다 |
| B. 책임 분리 관점 | `job`, `event`, `result`, `race_result`가 각자 다름 | 구조가 복잡해 보인다 | 역할이 섞이면 시스템이 무엇을 믿어야 할지 잃는다 |

핵심은 이렇다.

- 데이터를 나누는 이유는 보기 좋으라고가 아니라 **책임을 분리하기 위해서**다.
- 데이터가 꼬인다는 것은 값이 단순히 이상해진다는 뜻이 아니라, **의미와 신뢰도가 섞이는 상태**를 뜻한다.
- 시스템이 커질수록 "같은 데이터처럼 보이는 것들"을 구분하지 않으면, 기능 추가가 곧 불안정성 증가로 이어진다.

---

## 배경

이 프로젝트에서 처음 헷갈리기 쉬운 점은, 전부 "기록 관련 데이터"처럼 보인다는 것이다.

- 대회를 스크래핑한 결과
- 개인 검색으로 찾은 결과
- 수동 입력으로 만든 대회
- 확정된 경기 기록

겉보기에는 모두 비슷하다.
하지만 시스템 입장에서는 서로 다른 질문에 답하는 데이터다.

- `job`: 지금 무슨 작업이 진행 중인가?
- `event`: 어떤 대회 맥락에 속하는가?
- `result`: 후보가 무엇이었는가?
- `race_result`: 최종 확정 기록이 무엇인가?

즉 이들은 모두 "기록 관련"이지만, **같은 책임을 지지 않는다.**

---

## A 상세 설명: 한 덩어리 데이터 관점

### 비유

처음 보면 병원 문서가 다 비슷해 보인다.

- 접수 기록
- 검사 요청서
- 검사 결과 초안
- 최종 판독
- 처방전

모두 환자와 관련된 문서다.
그래서 초심자는 "한 파일에 다 넣어도 되는 것 아닌가?"라고 생각하기 쉽다.

하지만 그렇게 하면 큰일 난다.

- 접수 정보가 진단처럼 보이고
- 임시 검사 결과가 최종 판정처럼 보이고
- 예전 처방이 현재 처방처럼 남는다

이런 식으로 **문서의 역할이 섞이는 순간** 사고가 난다.

### 이 관점의 특징

- 비슷한 내용은 한 군데 섞여 있어도 괜찮다고 느낀다
- 저장만 잘 되면 문제없다고 생각한다
- 조회 시점의 의미 차이를 놓친다

이 프로젝트에 대입하면 이런 오해가 생긴다.

- `job`도 기록이니까 기록처럼 써도 되겠지
- `scrape_jobs.results`도 결과니까 SSOT처럼 써도 되겠지
- `member_search_jobs` 결과도 기록을 담고 있으니 최종 데이터처럼 보여도 되겠지

문제는 이런 사고방식이 처음엔 빨라 보여도, 나중에는 시스템이 스스로 모순에 빠지게 만든다는 점이다.

## B 상세 설명: 책임 분리 관점

### 비유

책임 분리 관점은 창고 운영과 비슷하다.

- `입고 대기` 구역
- `검수 중` 구역
- `불량/보류` 구역
- `출고 확정` 구역

모두 같은 상품처럼 보여도, 같은 랙에 두지 않는다.
왜냐하면 각 구역의 **의미와 허용 작업이 다르기 때문**이다.

이 프로젝트도 마찬가지다.

### 1. `job`은 과정 데이터

`scrape_jobs`, `member_search_jobs`는 "무슨 작업이 일어나고 있는가"를 담는다.

- 상태
- 진행률
- 중간 결과
- 검토 전 원본

즉 이것은 **진행 중인 과정**이다.

### 2. `event`는 대회 맥락 데이터

이벤트는 "어떤 대회인가"라는 문맥을 제공한다.

- 대회명
- 날짜
- source/sourceId
- 이 이벤트와 연결된 작업

즉 이벤트는 기록이 아니라 **기록이 매달리는 배경**이다.

### 3. `result`는 후보 데이터

검색이나 스크래핑 결과는 후보일 뿐이다.

- 오탐이 있을 수 있음
- 동명이인이 섞일 수 있음
- 검토 전임

즉 `result`는 **진실이 아니라 후보**다.

### 4. `race_result`는 확정 데이터

이건 최종적으로 믿는 기록이다.

- 최종 조회
- PB 계산
- 회원 기록 타임라인

즉 이 컬렉션은 **SSOT**다.

### 요약

| 데이터 | 역할 | 믿을 수 있는 정도 | 잘못 쓰면 생기는 문제 |
|---|---|---|---|
| `job` | 작업 상태 | 낮음 | 과정 데이터를 최종 데이터처럼 사용 |
| `event` | 대회 맥락 | 중간 | 같은 대회를 여러 문서로 중복 표현 |
| `result` | 후보 | 낮음 | 검토 전 후보를 확정처럼 사용 |
| `race_result` | 확정 기록 | 높음 | SSOT가 아닌 것을 SSOT처럼 사용 |

## "꼬인다"는 게 정확히 무엇인가

"꼬인다"는 표현은 막연하지만, 실제로는 네 가지가 섞이는 상태를 뜻한다.

### 1. 의미가 꼬임

예:

- 작업 문서를 기록 문서처럼 사용
- 후보 결과를 확정 기록처럼 사용

즉 **문서가 원래 답해야 할 질문이 흐려진 상태**다.

### 2. 신뢰도가 꼬임

예:

- `running` 상태 데이터를 최종 기록처럼 믿음
- 수동 입력과 스크랩 원본을 같은 수준으로 취급

즉 **무엇을 지금 믿어도 되는지 알 수 없는 상태**다.

### 3. 연결이 꼬임

예:

- `race_results.jobId`와 이벤트 doc ID가 불일치
- manual 이벤트와 scrape 이벤트가 같은 현실 대회인데 따로 남아 있음

즉 **데이터는 있어도 관계가 끊기거나 중복된 상태**다.

### 4. 책임이 꼬임

예:

- 공개 페이지가 운영 데이터까지 받아감
- 검색 작업이 최종 조회의 근거가 되어버림

즉 **어디까지가 공개용이고 어디까지가 운영용인지 흐려진 상태**다.

## 꼬이면 왜 무서운가

### 1. 화면마다 서로 다른 진실을 말할 수 있다

한 화면은 `race_results`를 보고,
다른 화면은 잘못 연결된 `job`이나 후보 데이터를 볼 수 있다.

그러면 사용자는 "기록이 있다/없다"를 화면마다 다르게 보게 된다.

### 2. 버그가 아니라 해석 불가 상태가 된다

코드 버그는 고치면 끝나는 경우가 많다.
하지만 데이터가 꼬이면 이런 질문에 답하기 어려워진다.

- 이 값이 최신인가?
- 원본인가 확정본인가?
- 수동 입력에서 온 건가?
- 스크랩에서 온 건가?
- 원래 이 문서가 어떤 책임을 갖고 있었나?

즉 **복구 비용이 급격히 올라간다.**

### 3. 기능을 추가할수록 증폭된다

처음엔 작은 중복처럼 보여도,

- 개인 검색
- 재스크래핑
- 캐시
- privacy
- 운영진 검토

같은 기능이 올라갈수록 같은 꼬임을 여러 곳이 증폭시킨다.

즉 나중에는 "새 버그"가 아니라 **기존 꼬임이 새 기능마다 다른 형태로 나타나는 상태**가 된다.

## 핵심 차이 (Decision Tree)

```text
이 데이터는 최종 진실인가?
├─ 예 → SSOT에 둔다
└─ 아니오
   ├─ 작업 상태를 담나?
   │  └─ 예 → job 데이터다
   ├─ 대회 맥락을 담나?
   │  └─ 예 → event 데이터다
   ├─ 후보/원본 결과를 담나?
   │  └─ 예 → result 데이터다
   └─ 공개 화면이 꼭 알아야 하나?
      ├─ 아니오 → 공개 응답에서 제거
      └─ 예 → 최소 정보만 전달
```

이 트리가 중요한 이유는, 대부분의 꼬임이 결국 이 질문을 생략해서 생기기 때문이다.

> "이 데이터는 도대체 무슨 책임을 가진 데이터인가?"

이 질문을 안 하고 저장/조회하면, 데이터는 금방 섞인다.

## 내 케이스 vs 상대 케이스 비교

### 내 케이스: 동호회 기록 시스템

- 작업 데이터와 확정 데이터가 함께 존재
- 공개 조회와 운영 화면이 함께 존재
- 수동 입력과 스크랩이 같이 존재
- 후보 결과를 사람이 확정하는 흐름이 존재

이 구조에서는 데이터를 구분하지 않으면 거의 반드시 꼬인다.

### 상대 케이스 1: 단순 소개 페이지

- 상태 없음
- 확정/후보 구분 없음
- 과정 데이터 없음

이 경우는 데이터 구분 부담이 훨씬 적다.

### 상대 케이스 2: 내부 관리자 전용 CRUD

- 공개 API 경계가 약함
- 수동 입력만 있고 외부 스크랩이 없을 수 있음

이 경우도 현재 프로젝트보다는 데이터 꼬임 위험이 낮다.

### 왜 내 케이스가 더 어렵나

이 프로젝트는 작아 보여도 실제로는 다음을 다 갖고 있다.

- 원본
- 후보
- 확정본
- 공개/비공개 경계
- 외부 의존성
- 수동/자동 입력 혼재

즉 **데이터 역할이 섞이기 쉬운 구조**다.

## 교훈

데이터를 구분해서 생성하고 유지하는 이유는 단순하다.

> **시스템이 무엇을 믿어야 하는지, 무엇을 아직 믿으면 안 되는지 구분하기 위해서다.**

그래서 앞으로는 새로운 기능을 볼 때 이렇게 생각하면 된다.

1. 이 데이터는 **과정**인가, **결과**인가?
2. 이 데이터는 **후보**인가, **확정본**인가?
3. 이 데이터는 **공개용**인가, **운영용**인가?
4. 이 데이터는 **맥락**을 담나, **진실**을 담나?
5. 이 데이터가 꼬이면 어떤 화면이 서로 다른 진실을 말하게 되나?

한 문장으로 줄이면 이렇다.

> 데이터가 꼬인다는 것은 값이 깨지는 게 아니라, **시스템이 데이터의 역할과 신뢰도를 더 이상 구분하지 못하게 되는 상태**다.
