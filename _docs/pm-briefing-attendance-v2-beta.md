# 출석 개선 베타 — PM용 설명서

> 대상: 비개발 운영·기획 담당자  
> 목적: **무엇이 바뀌었는지**, **회원에게 어떻게 보이는지**, **베타 운영 시 주의할 점**을 공유하기 위함  
> 관련 스펙: `_docs/superpowers/specs/2026-04-17-attendance-page-redesign-v2.md`, `2026-04-17-attendance-page-tech-spec.md`

---

## 1. 한 줄로 요약

**QR로 쓰는 기존 출석 페이지(`index.html`)는 그대로 두고**, 소수 테스트용으로 **새 주소(`attendance-v2.html`)**만 열었습니다. 새 페이지는 **정회원 명단에서 본인을 고른 뒤 한 번에 출석**할 수 있고, **같은 날 같은 사람이 두 번 출석하는 것은 서버에서 막습니다.** 게스트 출석도 같은 화면에서 할 수 있습니다.

---

## 2. 왜 이렇게 나눴나요? (베타 전략)


| 구분             | 설명                                                      |
| -------------- | ------------------------------------------------------- |
| **기존 (QR·메인)** | 지금까지 쓰던 방식 그대로. 닉네임·팀·정모 유형을 단계적으로 입력.                  |
| **베타 (새 URL)** | 운영진이 지정한 사람에게만 URL을 알려 줘서 써 보게 함. 문제가 있어도 QR 유입은 영향 없음. |


즉, **운영 리스크를 줄이면서** 새 UX를 검증하는 구조입니다.

---

## 3. 회원 입장에서 무엇이 달라지나요?

### 3.1 처음 베타 URL을 연 경우

1. **정회원 명단**이 불러와지고, 검색으로 본인을 찾습니다.
2. 본인을 누르면 이 브라우저(기기)에 **“나는 이 닉네임·이 회원 ID”**가 저장됩니다. (다음부터 검색 생략)
3. 화면에는 **오늘(또는 해당 요일에 맞는) 정모 날짜·유형**이 자동으로 잡히고, **출석 체크** 한 번으로 끝납니다.

### 3.2 두 번째부터 (같은 기기·브라우저)

- 저장된 프로필이 있으면 **검색 없이** 바로 출석 화면이 나옵니다.  
- **팀 변경**은 화면에서 바꿀 수 있으며, **다음 출석부터** 그 팀으로 기록됩니다. (과거 출석 기록을 바꾸지는 않습니다.)

### 3.3 게스트

- “게스트로 출석”으로 닉네임·날짜·정모 유형만 넣어 등록합니다.  
- **정책상** 게스트는 **같은 날 여러 번** 등록될 수 있게 두었습니다. (스펙: 게스트는 중복 출석 검사 제외)

### 3.4 “다른 사람으로 출석”을 누른 경우

- 이 기기에 저장된 프로필이 지워지고, 다시 **명단에서 검색**하는 흐름으로 돌아갑니다. (자봉이 여러 명을 찍을 때 등)

---

## 4. 서버·데이터 쪽에서 무엇이 바뀌었나요? (개념만)

### 4.1 출석 한 건에 붙는 정보

기존에도 있던 것: 닉네임, 팀, 정모 유형, 날짜, 시간 등.

**새로 저장할 수 있는 것**


| 항목                     | 의미                                                         |
| ---------------------- | ---------------------------------------------------------- |
| **회원 ID (`memberId`)** | Firestore `members` 컬렉션의 문서 ID. 정회원이 새 화면으로 출석할 때 함께 보냅니다. |
| **게스트 여부 (`isGuest`)** | 게스트 출석이면 `true`. 정회원이면 `false`.                            |


**기존 QR 페이지**는 예전처럼 `memberId` 없이도 출석됩니다. 그때는 `memberId`가 비어 있는 채로 저장됩니다(레거시와 호환).

### 4.2 “같은 날 두 번 출석” 막기

- **정회원(게스트 아님)**  
  - 닉네임(검색용 소문자 키) + 날짜가 이미 있으면 막습니다.  
  - 새 화면에서 **회원 ID까지**내면, **회원 ID + 날짜**로도 한 번 더 확인합니다. (닉네임을 바꾼 뒤에도 같은 날 이중 출석 방지에 도움)
- **게스트**  
  - 위와 같은 “하루 한 번” 규칙을 **적용하지 않습니다** (스펙대로).

### 4.3 새로 조회할 수 있는 것


| 기능          | 설명                                                                                 |
| ----------- | ---------------------------------------------------------------------------------- |
| **정회원 목록**  | 새 출석 화면의 검색용. 숨김 처리된 회원은 목록에 안 나옵니다.                                               |
| **이번 달 통계** | 출석 직후 화면에 “이번 달 몇 번 왔는지, 출석률, 연속으로 화·목·토 정모에 몇 번 나왔는지” 등을 보여 줍니다. (회원 ID가 있을 때 위주) |


출석률·연속 정모의 **계산 방식**은 개발 구현과 긴 테크 스팅 사이에 일부 표현 차이가 있을 수 있습니다. 숫자가 이상하면 개발에 로그·정의 확인을 요청하면 됩니다.

## 5. 배포·운영 시 PM이 알면 좋은 것

1. **순서**
  - DB 인덱스 반영 → 서버(Functions) 배포 → 웹 호스팅에 `attendance-v2.html` 올리기  
  - 인덱스가 준비되기 전에는 일부 기능이 느리거나 오류 날 수 있음.
2. **베타 URL 예시** (실제 도메인은 배포 환경에 따름)
  - `…/attendance-v2.html`  
  - QR(`index.html`)과 **주소가 다름**을 안내 문구에 적어 두는 것이 좋습니다.
3. **피드백 수집**
  - “검색이 불편하다”, “날짜가 어긋난다”, “통계 숫자가 이상하다” 등 **현상 + 스크린샷 + 사용 기기**를 받으면 개발이 재현하기 쉽습니다.

## 6. 아직 스펙/문서와 100% 같지 않을 수 있는 부분

개발 리뷰에서 나온 **정리가 필요한 지점**입니다. PM이 “알고만 있으면 되는 것”과 “제품 결정이 필요한 것”을 나눴습니다.


| 주제                   | 알고 있기                                                         | 결정이 필요하면                    |
| -------------------- | ------------------------------------------------------------- | --------------------------- |
| 정회원에 **회원 ID 필수** 여부 | 테크 스팅에는 “필수”로 적혀 있으나, **기존 QR을 살리려면** 예외가 있음                  | 언제부터 QR도 회원 ID 연동을 강제할지 로드맵 |
| **출석률 분모**           | “달력상 이번 달 화·목·토 며칠인가” vs “실제로 클럽이 모였던 날 수” 등 정의가 다를 수 있음      | 대외·대내 설명 문구를 어떤 정의로 통일할지    |
| **연속 출석**            | 화·목·토 **정모 유형으로 저장된 기록**만 연속에 넣음. `기타`만 찍힌 날은 연속 계산에서 빠질 수 있음 | “기타 모임도 연속에 넣을지” 정책         |
| 동시에 두 번 눌렀을 때        | 이론상 아주 짧은 간격이면 중복이 생길 수 있음(희귀)                                | 수용 가능하면 모니터링만, 아니면 개선 백로그   |
