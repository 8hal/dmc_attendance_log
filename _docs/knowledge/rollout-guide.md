# 롤아웃 가이드

> 2026-03-23 BETA 런칭 경험에서 추출한 교훈.
> 이 서비스뿐 아니라 소규모 서비스 일반에 적용 가능.

---

## 1. 배포는 쪼개라

한 번에 모든 것을 배포하면 문제 발생 시 원인 특정이 불가능하다.

| 단계 | 기준 | 예시 |
|------|------|------|
| 긴급 | 데이터 손상 가능한 버그 | confirmedCount 덮어쓰기 |
| 안전 | UI 변경, 신규 기능 (기존 기능 훼손 없음) | BETA 태그, 로깅, 네비게이션 |
| 정기 | 기존 동작 변경, 인증 방식 전환 | 비밀번호 서버검증, 검색 일원화 |

**원칙:**
- functions와 hosting은 별도 배포 (`--only functions`, `--only hosting`)
- functions 먼저, hosting 나중 (API 호환성)
- 데이터 마이그레이션은 백업 후 별도 단계

**안전 판단 기준:**
- "이 변경이 기존에 되던 것을 안 되게 만드는가?" → Yes면 정기 배포
- "이 변경이 없어도 서비스가 돌아가는가?" → Yes면 안전 배포

## 2. 로깅 없이 런칭하지 마라

BETA 런칭 첫날, 로깅이 없었다면:
- 19명이 접속한 것을 몰랐을 것
- 검색 전환율 18%라는 사실을 몰랐을 것
- "잘 되고 있겠지"라고 착각했을 것

**최소 로깅 셋:**

```
page_load        — 누가 언제 왔는지
[핵심액션]_start  — 시도했는지
[핵심액션]_complete — 성공했는지
[핵심액션]_error   — 왜 실패했는지
feedback         — 사용자 직접 의견
```

**이탈 감지 필수:**
- `search_cancel` — 어느 단계에서 포기했는지 (step 포함)
- `search_no_result` — 결과가 없어서 이탈한 건지
- `search_timeout` — 느려서 이탈한 건지

**로깅 타이밍:** 기능 개발과 동시에. 배포 후 추가하면 첫날 데이터를 놓친다.

## 3. 페르소나별 화면을 분리하라

같은 페이지에 모든 역할의 기능을 넣으면:
- 운영진이 개발자 도구를 보고 혼란
- 개발자 정보가 일반 사용자에게 노출

| 역할 | 페이지 | 접근 |
|------|--------|------|
| 비회원/회원 | my.html, races.html, index.html | 공개 |
| 운영진 | report.html, admin.html | 비밀번호 |
| 개발자 | ops.html | URL 직접 접근 |

**원칙:**
- 데이터 정합성, 이벤트 로그 = 개발자 전용
- 대회 수집/관리, 회원 관리 = 운영진 전용
- 기록 조회, 검색, 확인 = 회원/비회원 전용

## 4. SSOT를 지켜라, denormalized는 의심하라

`confirmedCount`(denormalized) ≠ 실제 `race_results` 수(SSOT)인 경우가 41건 발생.

**교훈:**
- denormalized 필드는 편의를 위한 캐시일 뿐, 진실이 아니다
- SSOT와 비교하는 정합성 체크를 반드시 만들어라
- 불일치가 발생해도 SSOT가 무사하면 서비스는 정상
- 불일치 알람은 심각도를 구분하라 ("41건 불일치" ≠ "41건 데이터 손실")

**재계산 가능하게 설계:**
```
confirmedCount = COUNT(race_results WHERE jobId = X AND status = confirmed)
```
언제든 SSOT에서 재계산할 수 있으면 denormalized 불일치는 공포가 아니라 정리 대상이다.

## 5. 유입보다 안정화가 먼저다

BETA 런칭일에 19명이 들어왔다. 전환율 18%.

**잘못된 순서:** 더 많은 사람 데려오기 → 같은 이유로 이탈 → 나쁜 첫인상 고착
**올바른 순서:** 이탈 원인 파악 → 수정 → 전환율 확인 → 그 다음 유입

**구체적 기준:**
- 검색→저장 전환율 50% 이상이 되기 전에는 홍보하지 않는다
- 전환율이 안 오르면 데이터 로그를 먼저 본다
- 데이터로 설명 안 되면 그때 설문한다 (3문항 이내, 단톡방에서)

## 6. 스크래퍼 외부 의존성을 과신하지 마라

외부 API는 예고 없이 바뀐다.

| 사례 | 원인 | 결과 |
|------|------|------|
| 인천하프 미수집 | myresult API 페이지네이션 미처리 | 대회 자체를 발견 못함 |
| 춘천마라톤 미발견 | 동일 원인 | 주요 대회 누락 |
| 엠제이 성별 오류 | 스크래퍼 성별 추론 로직 한계 | 랭킹에서 잘못 분류 |

**방어 원칙:**
- 외부 API 응답은 항상 검증 (빈 배열, 예상 밖 포맷)
- 페이지네이션이 있을 수 있다고 가정
- 스크래퍼가 추론한 값(성별 등)보다 내부 데이터(회원 프로필)를 신뢰
- 자동 수집 실패 모니터링 — 주요 대회가 누락되면 알 수 있어야 함

---

## 7. 삭제 기능은 반드시 필요하다

"잘못 입력은 없겠지"는 착각이다. 런칭 첫날 삭제 3건 발생.

**설계 원칙:**
- 삭제 버튼은 있되, 눈에 띄지 않게 (실수 방지)
- 본인 확인 절차 필수 (실명 입력 등)
- 삭제 로그 남기기 (누가, 무엇을, 언제)
- 로그인 없는 서비스에서의 인증은 "최선의 노력" 수준으로 타협

---

## 8. 피드백 채널은 마찰 최소화

구글폼, 별도 페이지 → 20명 규모에서 응답률 극히 낮음.

**효과적인 방법:**
- 페이지 하단 "불편한 점이 있나요?" 버튼 → `prompt()` → 로그 저장
- 이벤트 로그에 `feedback` 이벤트로 기록 → ops.html에서 조회
- 추가 도구/가입 없이 즉시 전송
