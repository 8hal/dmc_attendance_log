## 배번 양도 기능 버그 수정 (v0.14.1)

### 배경
4월 18-19일에 구현된 배번 양도 기능에서 연속된 양도 시도 시 잘못된 회원이 선택되는 버그 발견.
구체적으로 "다니엘 → 마운틴" 양도 후 "바람 → 데이" 양도 시도 시 "바람 → 마운틴"으로 표시됨.

### 문제 원인
- `renderTransferList` 함수에서 `member.memberId`를 사용했지만, `action=members` API는 `member.id`로 반환
- 결과: `memberId`가 빈 문자열(`""`)로 설정되어 `findIndex`가 항상 첫 번째 빈 항목(이전 양도로 추가된 비참가자) 반환

### 해결 방법
```javascript
// renderTransferList 함수 수정
const memberId = member.id || member.memberId;  // API 응답의 'id' 우선 사용
```

### 배포
- 디버그 로그 제거 완료
- 코드 정상 상태 복원

- 수동 수정 시 동기화 불가
- 신뢰도 불명확

### 설계 개요

**Medallion Architecture 도입**:
```
Bronze (scrape_jobs) - Raw 원본
    ↓
Silver (participants) - Staging
    ↓
Gold (race_results) - SSOT
```

**핵심 원칙**:
1. 단방향 흐름 (Bronze → Silver → Gold)
2. Gold 우선 (Gold 존재 시 Silver/Bronze 무시)
3. Silver 동기화 (Gold 생성/수정 시 자동 참조)
4. Bronze 불변 (생성 후 수정 불가)

### 5단계 구현 계획

| Phase | 작업 | 우선순위 |
|-------|------|----------|
| Phase 1 | Gold 메타데이터 (`dataLineage`, `trustLevel`) | P0 |
| Phase 2 | Silver 참조 (`confirmedResult`) | P0 |

## 메모

### 교훈

**1. 정책은 실제 구현을 따라야 함**
- 코드를 확인하지 않고 정책을 세우면 불일치 발생
- "DNS/DNF는 생성 안 함" → 실제로는 생성 중이었음

**2. 데이터 신뢰도 계층화 필요**
- Medallion Architecture로 Bronze/Silver/Gold 구분
- 수동 수정 시 동기화 문제 해결

**3. 정규화 함수의 중요성**
- `normalizeRaceDistance()` 일관 적용 필요
- 대소문자, 공백 등으로 매칭 실패 가능

**4. 서브에이전트 활용**
- 복잡한 디버깅은 서브에이전트로 병렬 분석
- 근본 원인 파악 시간 단축

**5. 사용자 요구사항 명확화**
- "DNS 처리"가 "DNS로 변경"인지 "DNS 생성"인지 명확히
- 오해로 인한 재작업 방지
