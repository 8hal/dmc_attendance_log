# 2026-04-10 (금요일) 주간 회고

## 요약

단체 대회 기능 UX 개선 및 삭제 기능 추가. 배포 후 도메인 지식 부족으로 인한 버그 발생 → 회고를 통해 개선 방안 수립.

## 주요 작업

### 1. 참가자 검색 Enter 선택 (group.html)
- 검색 후 Enter 키로 첫 번째 결과 즉시 선택
- 한글 IME 조합 중 Enter 무시 (compositionstart/end)
- 중복 선택 방지 (isEnterProcessing 플래그)
- 검색 필터링 복원 (input 이벤트 리스너)

**버그:**
- Enter 키 2명 선택: `e.stopPropagation()` + 플래그로 해결
- 잘못된 사람 선택: IME composing 무시로 해결
- 검색 필터링 안 됨: input 리스너 복원으로 해결

### 2. 저장/스크랩 분리 (ops.html)
- "저장 + 스크랩" → "저장", "스크랩" 2개 버튼으로 분리
- 소스 저장 전까지 스크랩 버튼 비활성화 (disabled, 회색)
- 사용자가 스크랩 타이밍 선택 가능

### 3. 고러닝 예정 대회 개선 (ops.html)
- 과거 대회 필터링 (오늘 이후만 표시)
- 검색창 + 월별 필터 + 더보기 버튼 추가 (20개씩 로드)
- 섹션 이름: "고러닝 예정 대회" → "예정 대회 (고러닝 + 러닝위키)"

### 4. 단체 대회 삭제 기능 (group.html + functions)
- 더보기 버튼(⋮) + 삭제 메뉴 추가
- 확인 다이얼로그 (참가자/기록 유지 안내)
- `group-events` API `subAction=delete` 구현

**버그:**
- 더보기 버튼 위치: `align-items: flex-start` → `center`
- **`apiBase is not defined`**: `apiBase()` → `API_BASE` (치명적 실수)

## 배포 이슈

### 문제 1: Functions 변경 감지 실패
- `firebase deploy --only functions`: "No changes detected"
- 해결: `firebase deploy --only functions:race` (특정 함수 타겟)

### 문제 2: API 요청 한도 초과
- 429 Error: `cloudbilling.googleapis.com` 분당 요청 한도 초과
- 원인: 짧은 시간에 너무 많은 배포 시도
- 해결: 1-2분 대기 후 재시도

### 문제 3: 브라우저 캐시
- 배포 완료 후에도 변경사항 미반영
- 해결: 하드 리프레시 (`Cmd + Shift + R`) 또는 시크릿 모드

## 근본 문제: AI 도메인 지식 부족

### 증상
1. **기본 패턴 미숙지**: `API_BASE` vs `apiBase()` 혼동
2. **서브에이전트 미활용**: 코드 리뷰만 1회 사용, 탐색/구현 에이전트 전혀 안 씀
3. **저성능 모델 사용**: `fast` 모델로 코드 리뷰 → 품질 낮음
4. **도메인 지식 축적 부재**: 같은 실수 반복

### 근본 원인 분석

**1. 프로젝트 패턴 문서화 부재**
- API 호출 패턴, 변수명 규칙 등 암묵지
- AI가 매번 추론해야 함 → 실수 발생

**2. 서브에이전트 활용 스킬 부족**
- `explore` 에이전트: 기존 코드 패턴 탐색 (안 씀)
- `generalPurpose`: 복잡한 구현 위임 (안 씀)
- `code-reviewer`: 저품질 모델 사용 (fast)

**3. 검증 프로세스 미흡**
- 구현 전 기존 코드 탐색 생략
- 배포 전 로컬 테스트 생략
- 코드 리뷰 1회만 수행 (배포 전 재검증 안 함)

## 개선 방안

### A. 프로젝트 지식 문서화 (단기)

**1. API 패턴 문서 생성**
```
_docs/development/api-patterns.md
- API_BASE 상수 사용 (함수 아님)
- fetch 패�n: POST는 body, GET은 query string
- 오류 처리: try/catch + showToast
```

**2. 공통 실수 문서**
```
_docs/development/common-mistakes.md
- apiBase() ❌ → API_BASE ✅
- curl_post for GET ❌ → curl_get ✅
- 한글 변수명 ❌ → ASCII ✅
```

**3. 변수명 규칙**
```
_docs/development/naming-conventions.md
- 상수: UPPER_SNAKE_CASE (API_BASE, IS_LOCAL)
- 함수: camelCase (loadGroupEvents, renderMemberList)
- DOM ID: kebab-case (member-filter, gorunning-list)
```

### B. 서브에이전트 활용 강화 (중기)

**1. 구현 전 탐색 필수화**
```
구현 시작 전:
1. explore 에이전트로 기존 패턴 탐색
   - "How are API calls made in this project?"
   - "What naming conventions are used?"
2. 패턴 문서 확인
3. 구현 시작
```

**2. 모델 선택 개선**
- `fast`: 단순 반복 작업만 (파일 복사, 간단한 수정)
- `default`: 코드 리뷰, 구현, 디버깅 (현재 미사용 → 활성화 필요)

**3. 서브에이전트 체크리스트**
```
복잡한 기능 구현 시:
☐ explore로 기존 패턴 확인
☐ generalPurpose로 구현
☐ code-reviewer로 리뷰 (default 모델)
☐ verification-before-completion으로 최종 검증
```

### C. 검증 프로세스 강화 (장기)

**1. 3단계 검증**
```
1. 구현 전: explore 에이전트로 패턴 탐색
2. 구현 후: code-reviewer로 코드 리뷰
3. 배포 전: verification-before-completion으로 최종 검증
```

**2. 로컬 테스트 자동화**
```
배포 전 체크리스트:
☐ 에뮬레이터에서 기능 테스트
☐ 브라우저 콘솔 오류 없음 확인
☐ 기존 기능 회귀 없음 확인
```

**3. 배포 후 검증**
```
배포 후:
☐ 시크릿 모드로 프로덕션 확인
☐ 주요 플로우 수동 테스트
☐ 브라우저 콘솔 오류 없음 확인
```
