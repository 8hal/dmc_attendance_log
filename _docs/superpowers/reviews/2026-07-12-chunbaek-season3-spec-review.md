## 1. 잘 잡힌 것 (유지 권장)

### 1.1 제품 포지셔닝

- 카카오톡 API 불가를 인정하고 **역할 분담**(단톡=소통, 앱=출석·집계)이 현실적
- DMC repo 포크가 아닌 **춘백 전용** 판단이 맞음
- 기성 솔루션(클루보, Strava 등) 조사 후 **100슬롯 커리큘럼**에 집중한 것이 타당

### 1.2 출석 도메인

- `isProgramOff` / 예외 / 출석 3단 분리 → 출정식 운영 규칙과 대응
- 예외는 **단톡 사전 알림 → 운영진만** 처리 → 패널티 회피 구조 차단
- 주 3회 = **훈련일 슬롯 중 (예외 제외) 출석 ≥ 3회** 확정

### 1.3 UX 핵심

```
매일:  오늘 → [출석하기] (10초)
동기:  팀 (풀 목표 + 주간 달성)
회고:  내 100일 (타임라인)
```

- 「체크인」이 아닌 **출석**, 홈이 **오늘** — 시즌 운영 문화와 일치

### 1.4 온보딩 (DMC 대비 차별점)

- 환영 → 명단 → **풀 목표(필수)** → 가이드 → 홈
- 출정식 각오를 프로필로 고정 → **팀 화면 목표 공개**와 연결
- 재로그인 시 프로필·가이드 스킵 → 100일 시즌에서 마찰 적음

### 1.5 Phase 로드맵

- MVP 출석 → Phase 2 km·몸무게 → Phase 3 공식 기록(PB/DNF)
- 온보딩에서 수집한 목표·PB가 Phase 3 **시드 데이터**로 활용 가능

## 2. Critical — 구현 전 반드시 정리

### C1. API 신원 검증 공백

**현상:** `my-profile`, 출석 저장 등이 클라이언트가 보낸 `memberId`에 의존. 링크만 있으면 타인 출석·조회 가능.

**리스크:** PRD의 「대리 출석 위험 낮음」 전제와 **기술적 보장이 불일치**.

**권장 (MVP 최소안):**

- `create-profile` / `link-device` 시 서버가 **opaque session token** 발급
- 이후 API는 `memberId` 직접 전달 대신 token 사용
- 또는 PRD에 **「신뢰 기반 내부 도구」** 명시 + 운영진 주간 감사 프로세스

**상태:** [ ] 미반영 (스펙 패치 필요)

---

### C2. 「N/100일」 vs 「출석 N일」 정의 미정

**현상:** 오늘 화면에 `42/100일`과 `출석 68일`이 공존. 의미가 다를 수 있음.

| 지표 | 추정 의미 | 제안 라벨 |
|------|----------|----------|
| 42/100 | 시즌 **경과 일차** (달력) | `42일차` |
| 68 | **출석한 훈련일 수** | `출석 68회` |

**권장:** 스펙 §7.1에 라벨·도움말 정의 고정.

**상태:** [ ] 미반영

---

### C3. §4.2 Out of Scope 표 오류

**현상:** 「개인 프로필 목표·PB 입력」이 Out of Scope 표에 있으나, 실제로는 **MVP 온보딩 In Scope** (§4.1, §16).

**권장:** §4.2 해당 행 삭제 또는 「Phase 3 공식 기록 집계」와 구분하여 수정.

**상태:** [ ] 미반영

---

### C4. §9.1 `members` 스키마 불일치

**현상:** §9.1 `members`에 `profileComplete`, `goalMarathonNetTime` 등 없음. §15.8에만 존재.

**권장:** §9.1을 §15.8 기준으로 통합.

**상태:** [ ] 미반영

## 3. Important — 시즌 운영 중 마찰 가능

### I1. 주 3회 달성 불가능 주차

한 주 **훈련일 슬롯이 3일 미만**이면 주 3회 목표 달성 불가.

**권장:**

- 100슬롯 import 시 **주당 훈련일 ≥ 3** 검증
- 또는 `weeklyTarget = min(3, 해당주 훈련일 수)` 자동 조정

**상태:** [ ] 미결정

---

### I2. 과거 출석 소급 정책

§8.2 「빠진 날 보완」은 있으나 **기한·승인** 미정.

**권장 (MVP):** 해당 **주 일요일 23:59**까지 본인 소급. 이후는 운영진 대리 입력.

**상태:** [ ] 미결정

---

### I3. 시즌 집계와 예외

주간: 예외는 분모·분자 제외 (명확).  
시즌 %: §3.3은 「출석한 슬롯 수」만 기술 — **예외·프로그램 휴무일 분모 처리** 문장 추가 필요.

**상태:** [ ] 미반영

---

### I4. 재로그인 시 ① 환영 매번

새 기기마다 환영 → 명단까지 번거로울 수 있음.

**권장:** `profileComplete` 재로그인은 **② 명단부터** 시작 (환영 스킵 옵션).

**상태:** [ ] 미결정

---

### I5. 4주 연속 미달 · 패널티

출정식 규칙은 있으나 앱 미반영 (운영진 수동). MVP로 Acceptable.

**권장:** 운영 화면에 **「N주 연속 미달」** 필터 추가 (구현 비용 낮음, 효과 큼).

**상태:** [ ] Phase 1.5 후보

---

### I6. 팀 화면 와이어 숫자

와이어 `12/18명` vs 실제 **~40명**. 목업 수치만 불일치 — **검색 + 주차 필터** 필수 명시.

**상태:** [ ] 와이어 수정 권장

## 6. MVP 범위 적정성

| 포함 항목 | 판단 |
|----------|------|
| 100슬롯 + 출석 | ✅ 필수 |
| 온보딩 5단계 + 프로필 | ✅ 팀 목표 공개·가이드 효과. 1~2일 추가 개발 감수 가치 |
| 사진 선택 | ✅ 운영진 단톡 왕복 감소 |
| 팀·운영 화면 | ✅ 없으면 시트와 차별화 약함 |
| 마일리지·몸무게 | ✅ Phase 2 적절 |
| RSVP | ✅ 단톡 유지 적절 |

**범위 creep:** 온보딩 + 프로필 + 출석 + 운영 그리드 + 100슬롯 import → **2~3주 1인 개발** 현실적.

## 10. 리뷰 결론

기획 **방향은 Go**. 출석 코어, 단톡과의 공존, 온보딩을 통한 팀 목표 노출은 춘백 시즌3 운영과 잘 맞는다.

구현 착수 전 **Critical 4건**을 원본 PRD에 반영하면, MVP 구현 계획 단계로 넘어갈 수 있다.
