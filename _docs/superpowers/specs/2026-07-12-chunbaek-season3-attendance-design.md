## 1. 배경 및 문제

### 1.1 프로그램 개요

동마클 「춘천 마라톤 100일 준비방」의 **춘백 시즌3**는 100일 동안 함께 훈련하며 춘천마라톤·제주마라톤 완주를 목표로 하는 시즌제 프로그램이다.

**시즌 미션 (출정식 기준)**

1. 춘마·제마 DNF 0%
2. 전원 부상 없이 완주
3. PB 달성률 60% 이상
4. 새벽훈련 출석률 주 3회 이상 (토요일 동마클 훈련 인정)

**운영 규칙 (출석 관련)**

- 출석률은 1주 단위로 확인
- 주 3회 미달 시 다음 훈련에서 CU 커피·간식 준비 (패널티)
- 개인 사정은 사전에 관리자에게 알리면 예외 인정
- 4주 연속 미달 시 참여 여부 상의

### 1.2 현재 방식의 한계

| 방식 | 문제 |
|------|------|
| **카카오톡 단톡** | 일정·참석 투표는 충분하나, **주·월·시즌 통계**가 없음. 단톡 사진과 출석 집계가 분리됨 |
| **구글 시트** | 회원이 본인 행을 찾기 어렵고, 모바일 입력·수식 유지·개인 대시보드가 불편함 |
| **기성 앱 (클루보, Strava 등)** | 출석·RSVP는 되나 **100슬롯 커리큘럼 + 시즌 출석 집계**에 맞지 않음. 카카오톡 투표 API 연동 불가 |
| **DMC attendance log** | QR 정모 출석·대회 기록 SSOT에 최적화. **100일 시즌·슬롯 단위 출석**과는 도메인이 다름 |

### 1.3 카카오톡 연동 불가 (조사 결과)

- 단톡 일정·투표 결과를 읽는 **공식 API 없음** (개인정보 보호로 거절 사례 다수)
- 톡캘린더 API는 **개인 캘린더**용이며 단체방 일정과 무관
- PC에서 단톡 메뉴를 외부 웹으로 원격 제어하는 것도 **불가**
- **절충안**: 단톡은 소통·응원·사진 공유 유지, 숫자·출석은 **웹 링크**로 수집

### 1.4 최종 목표 (시즌 전체)

> 모든 참여자가 **원하는 기록을 내도록 돕는 것** — 궁극적으로는 춘마·제마 **공식 기록 (PB/DNF)** 이지만, 그 전제 조건으로 **꾸준한 훈련 출석**이 가장 중요하다.

**본 문서(MVP)는 출석에 한정한다.** 마일리지·몸무게·대회 기록은 Phase 2 이후.

## 3. 핵심 도메인 모델

### 3.1 기본 개념

```
시즌 (100일)
  └─ 슬롯 1 ~ 100   (각각 = 1일)
        ├─ 훈련 내용 (시즌 시작 전 운영진이 고정)
        └─ 출석 기록 (회원 본인 입력)
              ├─ 출석 여부
              ├─ 사진 (선택, 나중에 필수 전환 가능)
              └─ 메모 (선택)
```

### 3.2 출석 상태 정의

출석 관련 개념은 **슬롯 유형**(프로그램)과 **회원별 상태**(개인)로 나눈다.  
「휴식」이라는 말은 개인이 쉬는 것으로 오해될 수 있어, 프로그램 전체 휴무는 **프로그램 휴무일**로 부른다.

#### A. 슬롯 유형 (100슬롯, 운영진 사전 설정)

| 슬롯 유형 | 필드 | 의미 | 회원 행동 | 주간 집계 |
|----------|------|------|----------|----------|
| **훈련일** | `isProgramOff: false` | 그날 프로그램상 훈련이 있는 날 | 출석 표시 가능 | 분모에 포함 |
| **프로그램 휴무일** | `isProgramOff: true` | **전원** 훈련 없음 (커리큘럼상 off) | 아무 것도 안 함 | **제외** (그날 출석 의무 없음) |

> **개인이 그날 안 뛰는 것**은 프로그램 휴무가 아니다. 훈련일인데 참여하지 않으면 **미출석**이며, 사전 승인된 사정은 운영진이 **예외**로 처리한다.

#### B. 회원별 상태 (훈련일 슬롯만)

| 상태 | 의미 | 누가 설정 | 주 3회 집계 |
|------|------|----------|------------|
| **미표시** | 아직 안 함 / 불참 | — | 미달 가능 |
| **출석** | 그날 훈련에 참여 (새벽·동마클 토요일·개인 훈련 포함) | **회원** (`[출석하기]`) | +1 |
| **예외** | 사전 승인된 불참 (출장·건강·경조사 등) | **운영진만** | **제외** (분모·분자 모두) |

**회원 앱에 예외 버튼 없음** — 주 3회 회피를 막기 위함.

#### C. 예외 처리 흐름 (확정)

```
회원: 단톡방에 사전 알림 ("이번 주 수요일 출장으로 불참")
  ↓
운영진: 단톡 확인 후 운영 화면에서 해당 회원·슬롯을 [예외] 처리
  ↓
시스템: 해당 슬롯은 주 3회 계산에서 제외
```

- 예외는 **출석이 아님** (`attended: false`, `exception: true`)
- 운영진은 필요 시 **출석 대리 입력**도 가능 (회원이 앱에서 깜빡한 경우)
- 예외 처리 시 메모 권장 (예: "출장, 단톡 4/10")

#### D. MVP에서 다루지 않는 것

| 개념 | MVP | 이후 |
|------|-----|------|
| **참석 예정** (RSVP) | Out of scope — 단톡 투표 유지 | Phase 1.5 검토 |
| **개인 휴무** | 별도 상태 없음 → 미출석 | — |

#### E. 주 3회 계산 (확정)

```
해당 주 훈련일 슬롯(isProgramOff === false) 중
  예외 슬롯 제외한 분모 대비
  출석(attended === true) ≥ 3  → 달성
```

프로그램 휴무일 슬롯은 애초에 출석 대상이 아니다.

### 3.3 집계

| 단위 | 계산 |
|------|------|
| **주간** | 해당 주 훈련일 슬롯 중 출석 횟수, 목표 3회 달성 여부 |
| **월간** | 해당 월 훈련일 슬롯 중 출석 횟수·출석률 |
| **시즌 출석 횟수** | 오늘까지 훈련일 슬롯 중 `attended === true` 건수 |
| **시즌 출석률** | 분모: 오늘까지 훈련일 슬롯 (`!isProgramOff`), 예외 슬롯은 분모·분자 모두 제외. 분자: 출석 건수 |
| **시즌 경과** | `dayIndex / 100` (달력 진행, 출석과 무관) |

집계는 `chunbaek_attendance` + `chunbaek_slots` 를 쿼리하여 계산한다. MVP에서 별도 집계 컬렉션은 두지 않는다.

## 4. 범위

### 4.1 MVP (In Scope)

- 100개 `chunbaek_slots` (날짜, 주차, 훈련 라벨, 프로그램 휴무 여부)

## 6. 정보 구조 (IA)

### 6.1 앱 진입 (라우팅)

**베이스 URL:** `https://dmc-attendance.web.app/chunbaek/` (§10)

```
/chunbaek/ 접속
  │
  ├─ localStorage.chunbaekSessionToken 있음
  │     └─ GET my-profile (token) → chunbaekS3.profileComplete true → 메인 앱 (오늘)
  │     └─ profileComplete false → 온보딩 ③ 프로필부터 재개
  │
  └─ token 없음 (첫 방문·새 기기·캐시 삭제)
        └─ 온보딩 ① 환영부터
              └─ ② 명단 선택 후
                    ├─ profileComplete false → ③ 프로필 → ④ 가이드 → 홈 (+ token 발급)
                    └─ profileComplete true  → ③④ 건너뜀 → 홈 (link-device + token 발급)
```

**재로그인**: 프로필이 이미 있는 참가자는 **프로필 생성·가이드를 다시 보지 않는다**. 명단에서 본인 확인만 하고 서버가 **session token**을 재발급한다.

**API 신원 (C1):** 출석·프로필 API는 클라이언트가 보낸 `memberId`를 신뢰하지 않는다. `create-profile` / `link-device` 성공 시 서버가 opaque token을 발급하고, 이후 요청은 `Authorization: Bearer <token>` 또는 `?token=` 으로 인증한다 (§9.4).

온보딩은 **하단 탭 없음**. `profileComplete` + token 저장 후 메인 IA 노출.

### 6.2 모바일 — 하단 탭 4개 (온보딩 완료 후)

```
[ 오늘 ]  [ 내 100일 ]  [ 팀 ]  [ 나 ]
```

| 탭 | 역할 |
|----|------|
| **오늘** | 오늘 슬롯 출석 (메인 활동, 매일 방문) |
| **내 100일** | 100슬롯 타임라인 + 주/월/시즌 출석 집계 |
| **팀** | 팀 시즌 진행률, 주차별 출석, **참가자 풀 목표 기록** |
| **나** | 닉네임, 시즌 출석 요약 (MVP 최소) |

### 6.3 운영진

- URL `/chunbaek/admin.html` (또는 비공개 경로 + 간단 인증)
- **PC 레이아웃** (출석 그리드)

### 6.4 단톡 연동 (비API)

- 단톡 고정 공지에 **「오늘 출석」웹 링크** 1개: `https://dmc-attendance.web.app/chunbaek/`
- 사진은 단톡에도 올릴 수 있으나, 앱에도 **선택** 업로드 가능 (운영진이 앱만 봐도 확인 가능하도록)

## 7. 화면 설계

### 7.1 오늘 (홈) — 메인 활동

가장 자주 쓰는 화면. 시즌 중 매일 이 화면으로 출석한다.

```
┌─────────────────────────────┐
│  춘백 시즌3   42일차 / 100일 │  ← 시즌 경과 (달력, §3.3)
│  출석 68회 · 출석률 68%      │  ← 훈련일 출석 횟수·률
├─────────────────────────────┤
│  42일차 · 4월 15일 (화)      │
│  📋 5km 인터벌 + 코어 20분   │  ← chunbaek_slots.trainingLabel
├─────────────────────────────┤
│                             │
│     [  ✓  출석하기  ]       │  ← 메인 CTA
│                             │
│  사진 (선택)  [ + 추가 ]    │
│  메모 (선택)  [        ]    │
├─────────────────────────────┤
│  이번 주  2 / 3회 출석      │
└─────────────────────────────┘
```

**동작**

- **출석하기** 탭 → `attended: true` 저장 → 토스트 (예: "42일차 출석 완료 · 이번 주 3/3")
- 이미 출석 → **출석 완료 ✓** 상태, 수정·사진 추가 가능
- **프로그램 휴무일** → "오늘은 프로그램 휴무일입니다" (출석 버튼 없음)
- 시즌 전/후 → 안내 메시지

**검증 (MVP)**

- `photoRequired: false` (기본) → 사진 없이 제출 가능
- 추후 `photoRequired: true` → 사진 없으면 제출 차단 (설정만 변경)

### 7.2 내 100일

주차별 접이 타임라인. 슬롯 탭 시 출석 폼(오늘 화면과 동일).

```
[ 주간 ] [ 월간 ] [ 시즌 ]   ← 집계 서브탭

▼ 7주차  4/7 ~ 4/13
   ●●○○○  2/3회 출석
   36일  인터벌    ✓  📷
   37일  (휴무)    —
   38일  장거리    ✓
   39일  —         미출석
   ...
▶ 6주차
```

**표기**

- ● 출석 / ○ 미출석 / — 프로그램 휴무 / 🟡 예외
- 오늘 슬롯 강조

**주간 집계 예시**

```
7주차 요약
```

## 9. 데이터 모델

춘백 시즌3 데이터는 **기존 `dmc-attendance` Firestore**에 저장한다. DMC `members` 컬렉션을 재사용하고, 춘백 전용 필드는 `chunbaekS3` 중첩 객체와 `chunbaek_*` 컬렉션으로 분리한다 (§10).

### 9.1 `members` (기존 + 춘백 확장)

약 **40명** 참가자는 DMC 정회원 `members`의 **부분집합**이다. 온보딩 명단 API는 `chunbaekS3.participant === true` 이고 `hidden !== true` 인 문서만 반환한다.

```javascript
// 문서 ID = memberId (DMC 기존과 동일)
members {
  // ── DMC 기존 (변경 없음) ──
  nickname: "홍길동",
  realName: "홍길동",       // optional
  gender: "M",              // optional
  team: "1조",              // optional
  hidden: false,

  // ── 춘백 시즌3 전용 (중첩) ──
  chunbaekS3: {
    participant: true,       // 시즌 참가자만 true (운영진 사전 설정, ~40명)
    profileComplete: false,  // 온보딩 ③ 완료 시 true
    goalMarathonNetTime: 16200,   // 필수, 초 단위 (4:30:00)
    existingPbNetTime: 17520,     // 선택, 초 단위. 없으면 null
    profileCreatedAt: Timestamp,
    profileUpdatedAt: Timestamp,  // optional
    kakaoId: "",             // Phase 1.5: optional
  }
}
```

> **C4 통합:** 프로필·온보딩 필드는 루트가 아닌 `chunbaekS3` 아래에 둔다. DMC 출석·대회 기록과 충돌하지 않는다.

**시드 (시즌 시작 전):** 운영진이 참가 40명에 `chunbaekS3.participant: true` 를 설정한다. `profileComplete`는 가입 전 `false`.

### 9.2 `chunbaek_slots` (100건, 시즌 전 고정)

```javascript
chunbaek_slots/{slotId} {   // slotId = 1..100 (문서 ID 권장)
  dayIndex: 42,              // 1..100
  date: "2026-04-15",        // ISO date
  week: 7,                   // 주차
  trainingLabel: "5km 인터벌 + 코어 20분",
  trainingType: "interval",  // optional: interval | long | dawn | dongmak_sat | off
  isProgramOff: false        // true = 프로그램 휴무일 (전원 훈련 없음, 출석 집계 제외)
}
```

### 9.3 `chunbaek_attendance` (회원 × 슬롯)

문서 ID 권장: `{memberId}_{slotId}` (멱등 upsert)

```javascript
chunbaek_attendance/{memberId}_{slotId} {
  memberId: "...",
  slotId: 42,
  attended: true,            // exception이 true이면 false
  exception: false,          // 운영진만 true (단톡 사전 알림 확인 후)
  exceptionNote: "",           // 예: "출장, 단톡 4/10"
  exceptionSetBy: "",          // 운영진 memberId (optional)
  photoUrl: "",              // optional, Firebase Storage
  note: "",
  updatedAt: Timestamp,
  updatedBy: "member" | "admin"
}
```

### 9.4 `chunbaek_sessions` (API 신원 — C1)

MVP는 명단 선택이지만, **API는 session token으로 신원을 검증**한다 (리뷰 C1).

```javascript
chunbaek_sessions/{tokenId} {
  token: "opaque-random-string",  // 클라이언트에만 전달
  memberId: "...",
  createdAt: Timestamp,
  expiresAt: Timestamp,           // 예: +100일 (시즌 종료까지)
  revoked: false
}
```

**발급:** `create-profile`, `link-device` 성공 시 서버가 token 생성 → 응답 body + 클라이언트 `localStorage.chunbaekSessionToken` 저장.

**검증:** `my-profile`, `save-attendance` 등 회원 API는 token 필수. `memberId` 쿼리 파라미터만으로는 조회·저장 불가.

**운영진 API:** `verify-admin` (DMC 패턴) + admin 전용 action은 token 대신 admin 세션 사용.

### 9.5 `chunbaek_season_config` (단일 문서)

문서 ID: `chunbaek-s3` (고정)

```javascript
chunbaek_season_config/chunbaek-s3 {
  seasonId: "chunbaek-s3",
  title: "춘백 시즌3",
  startDate: "2026-...",
  endDate: "2026-...",
  weeklyTarget: 3,
  photoRequired: false       // true로 변경 시 사진 필수
}
```

### 9.6 Storage

```
chunbaek/attendance/{seasonId}/{slotId}/{memberId}.jpg
```

- 클라이언트 업로드 전 리사이즈 (긴 변 1200px, JPEG ~80%)

### 9.7 집계 쿼리 (개념)

- **주간 출석 수**: `chunbaek_attendance` where `slotId` in (해당 주 slots) and `attended == true`
- **주 목표 달성**: 출석 대상 슬롯(`!isProgramOff`, `!exception`) 대비 `attended === true` ≥ `weeklyTarget`
- **시즌 출석 횟수**: 오늘까지 훈련일 슬롯 중 `attended === true` 건수
- **시즌 출석률**: 분모 = 오늘까지 훈련일 슬롯 (`!isProgramOff`), 예외 슬롯은 분모·분자 모두 제외
- **시즌 경과**: `dayIndex / 100` (달력 진행, 출석과 무관)

## 10. 인프라·명단 연동

### 10.1 프로젝트·배포 (확정)

| 항목 | 결정 |
|------|------|
| **Firebase 프로젝트** | **별도 프로젝트 아님** — 기존 `dmc-attendance` (prod), `dmc-attendance-staging` (staging) |
| **Repo** | 동일 monorepo (DMC attendance log) |
| **Hosting URL** | `https://dmc-attendance.web.app/chunbaek/` |
| **정적 파일** | 루트 `chunbaek/` 디렉터리 (`index.html`, `admin.html`, …) |
| **Hosting `public`** | `.` (기존과 동일 — `/chunbaek/` 는 서브경로) |

DMC 메인 출석·대회 기록과 **같은 Firestore·같은 Functions 런타임**을 공유한다. 춘백만의 브랜딩·100슬롯 UI는 `/chunbaek/` 경로로 분리한다.

### 10.2 API (`/api/chunbaek`)

`race` API와 동일 패턴:

| 구성 | 값 |
|------|-----|
| Cloud Function | `exports.chunbaek` (`functions/index.js`) |
| Hosting rewrite | `/api/chunbaek` → `chunbaek` (region `asia-northeast3`) |
| CORS | `race`와 동일 (`Access-Control-Allow-Origin: *`) |

**클라이언트 API 베이스 (패턴):**

```javascript
const IS_LOCAL = location.hostname === "localhost" || location.hostname === "127.0.0.1";
const API_BASE = IS_LOCAL
  ? `http://${location.hostname}:5001/dmc-attendance/asia-northeast3/chunbaek`
  : "/api/chunbaek";
```

에뮬레이터: `firebase emulators:start` 후 `http://localhost:5000/chunbaek/` + functions 포트 5001.

### 10.3 명단 연동 (DMC `members` 부분집합)

```
DMC members (~정회원 전체)
  └─ chunbaekS3.participant: true  (~40명)
        └─ 온보딩 ② 명단 API 노출
              └─ profileComplete 후 출석·팀 집계
```

| 단계 | 작업 |
|------|------|
| 시즌 전 | 운영진이 참가 40명 `members` 문서에 `chunbaekS3.participant: true` 설정 (스크립트 또는 admin) |
| 온보딩 ② | `GET members-roster` → `participant` + `!hidden` 만 반환 |
| 신규 가입 | `POST create-profile` → `chunbaekS3.profileComplete: true`, 목표·PB 저장, **session token 발급** |
| 재로그인 | `POST link-device` → `profileComplete` 확인 후 **token 재발급** |
| DMC 연동 | 닉네임·성별 등은 기존 `members` SSOT. 춘백 전용 필드만 `chunbaekS3` |

**비참가 정회원:** `participant` 없음 또는 `false` → 춘백 명단·API 접근 불가.

### 10.4 `firebase.json` 추가 (구현 시)

```json
  "function": { "functionId": "chunbaek", "region": "asia-northeast3" }
}
```

## 15. 가입·로그인 (초안)

춘백은 **폐쇄형 시즌 프로그램**이다. 참가자 명단은 시즌 시작 전 확정되며, 공개 가입은 없다.

### 15.1 요구사항

| 항목 | 내용 |
|------|------|
| 대상 | 시즌 참가자만 (**약 40명**) |
| 목적 | 출석·사진이 **본인 계정**에 붙도록 식별 |
| UX | 모바일, 링크 1번으로 시작, 재방문 시 바로 **오늘** 화면 |
| 보안 | 대리 출석 가능성 **낮음** (단톡 폐쇄·상호 신뢰) — PIN·OAuth 필수 아님 |
| 운영진 | 회원 명단 사전 등록, admin 인증 별도 |
| 인프라 | **카카오 개발자 계정 보유** (단톡 API 불가, 로그인 API는 Phase 1.5에서 활용 가능) |

### 15.2 DMC 패턴 (참고)

DMC 출석은 **로그인 없음**:

1. `members` API에서 정회원 목록 로드
2. 닉네임 선택 → `localStorage`에 저장
3. 이후 원클릭 출석
4. 「다른 사람으로」로 프로필 초기화

**장점:** 10초 온보딩. **단점:** 링크만 있으면 남의 닉네임 선택 가능.

### 15.3 후보 방안

#### A. 명단 선택 + localStorage (DMC 동일)

```
첫 방문: 명단에서 본인 닉네임 선택 → 확인
재방문: 저장된 프로필로 자동 진입
```

| | |
|--|--|
| ✅ | 구현 최단, MVP에 적합 |
| ✅ | 폰 바꿔도 닉네임만 다시 선택 |
| ⚠️ | 링크 유출 시 대리 선택 가능 (소규모·단톡 폐쇄면 실무상 허용되는 경우 많음) |

#### B. 명단 선택 + 개인 PIN (추천 후보)

```
운영진이 회원별 4자리 PIN 사전 발급 (또는 최초 1회 본인 설정)
첫 방문: 닉네임 + PIN → 서버 검증 → 세션 저장
```

| | |
|--|--|
| ✅ | 구현 단순, 대리 선택 방지 |
| ✅ | 카카오 OAuth 없이 실명 수준 식별 |
| ⚠️ | PIN 분실 시 운영진 재설정 필요 |

#### C. 개인별 초대 링크

```
운영진이 회원마다 고유 URL 발급 (토큰 포함)
링크 클릭 = 본인으로 자동 로그인
```

| | |
|--|--|
| ✅ | UX 최고 (탭 없음) |
| ⚠️ | 인원수만큼 링크 관리·재발급 부담 |
| ⚠️ | 링크 전달 시 타인 사용 가능 |

#### D. 카카오 로그인

```
카카오 OAuth → members.kakaoId 와 매칭
```

| | |
|--|--|
| ✅ | 실제 계정 식별, 재설치 후에도 동일인 |
| ⚠️ | 카카오 개발자 앱·동의항목 설정 |
| ⚠️ | 최초 1회 운영진이 카카오ID↔명단 수동 매칭 또는 가입 시 선택 |
| ⚠️ | 단톡 API 연동은 여전히 불가 (로그인만 가능) |

#### E. 카카오 로그인 + 명단 매칭 (하이브리드)

```
카카오 로그인 후 "명단에서 본인 선택" 1회 → kakaoId 저장
이후 자동 로그인
```

### 15.4 역할별 인증 (운영진)

DMC `verify-admin` 패턴 권장:

- 회원 화면과 **분리**
- `POST verify-admin` + 비밀번호 (환경변수)
- `sessionStorage`에 인증 상태 저장
- admin URL은 단톡에 공개하지 않음

### 15.5 MVP 결정 (확정)

**A. 명단 선택 + localStorage** (DMC 동일)

대리 출석 위험이 낮고, 40명 규모에서 PIN·개인 링크 40개는 **운영 부담 대비 이득이 작다**. 카카오 개발자 계정이 있어도 MVP 일정을 위해 OAuth는 Phase 1.5로 미룬다.

```
[첫 방문 — 온보딩 5단계, §16]
1. 환영
2. 명단에서 닉네임 선택
3. 프로필 생성 (풀 목표 필수, PB 선택)
4. 출석 가이드·회칙
5. → 오늘 홈

[재방문]
localStorage.chunbaekSessionToken + profileComplete → 바로 오늘
```

| 판단 근거 | |
|----------|--|
| 대리 출석 낮음 | PIN(B)·개인링크(C) 생략 |
| 40명 | 명단 선택 UI로 충분 (검색 필드 권장) |
| 카카오 dev 계정 | MVP 필수 아님 → **Phase 1.5** 카카오 로그인 검토 |
| 100일 시즌 | 폰 교체 시 닉네임 재선택만 하면 됨 (불편하면 그때 카카오 추가) |

### 15.6 Phase 1.5 (선택) — 카카오 로그인

개발자 계정이 있으므로, MVP 이후 필요 시 **E안** 적용:

```
카카오 로그인 → (최초 1회) 명단에서 본인 선택 → kakaoId 저장
이후: 기기 바뀌어도 카카오로 자동 식별
```

- 단톡 방 API 연동은 여전히 불가
- `members.kakaoId` 중복 가입 방지 (1 카카오 = 1 명단)

### 15.7 채택하지 않는 안 (이번 시즌)

| 안 | 사유 |
|----|------|

| **C. 개인 링크 40개** | 운영·재발급 부담, 이득 적음 |
| **D. 카카오 only (MVP)** | 명단 매칭 1회는 필요, MVP 일정 지연 |

### 15.8 데이터 (인증·프로필)

## 16. 온보딩 — 가입 및 프로필 생성 (MVP, 신규)

DMC에는 없는 **전용 온보딩 플로우**. 시즌 참가자가 앱에 처음 들어올 때 1회 거친다.

### 16.1 플로우 개요

**신규 가입** (프로필 없음)

```
① 환영 → ② 명단 → ③ 프로필 → ④ 가이드 → ⑤ 홈
```

**재로그인** (`chunbaekS3.profileComplete === true`, token 없음)

```
① 환영 → ② 명단(본인 선택) → ⑤ 홈     ※ ③④ 생략
```

**일상 재방문** (token 있음)

```
⑤ 홈 (온보딩 전부 생략)
```

| 단계 | URL (예) | 신규 | 재로그인 |
|------|----------|------|----------|
| ① 환영 | `/chunbaek/onboarding.html` | ○ | ○ |
| ② 명단 | `/chunbaek/onboarding-pick.html` | ○ | ○ |
| ③ 프로필 | `/chunbaek/onboarding-profile.html` | ○ | **건너뜀** |
| ④ 가이드 | `/chunbaek/onboarding-guide.html` | ○ | **건너뜀** |
| ⑤ 홈 | `/chunbaek/` (오늘) | ○ | ○ |

### 16.2 ① 환영

```
┌─────────────────────────────┐
│                             │
│      춘백 시즌3              │
│   100일, 함께 달립니다       │
│                             │
│  출정식에서 정한 목표를       │
│  프로필로 남기고,            │

│                             │
│  [      시작하기      ]      │
│                             │
└─────────────────────────────┘
```

- 시즌명·기간(선택) 표시
- 로그인 없음 — 다음 단계에서 명단 매칭

### 16.3 ② 명단에서 닉네임 선택 (가입 · 재로그인 공통)

```
┌─────────────────────────────┐
│  당신은 누구신가요?          │
│  [🔍 이름 검색        ]      │
├─────────────────────────────┤
│  ○ 게살볶음밥               │
│  ○ 홍길동                   │
│  ○ 김러너        가입됨 ↩    │  ← 재로그인 가능
│  ○ 이페이스                 │
│  ... (40명)                 │
├─────────────────────────────┤
│  [      다음      ]          │
└─────────────────────────────┘
```

**규칙**

- 목록: `chunbaekS3.participant === true` 인 `members` (~40명, 가나다순)
- `chunbaekS3.profileComplete === false` → **신규 가입** → ③으로
- `profileComplete === true` → **재로그인** → ③④ 건너뛰고 ⑤ 홈 (`POST link-device` → token 발급)
- 검색: 닉네임 부분 일치
- 확인 모달:
  - 신규: **「김러너님 맞습니까? 프로필을 만들게요」**
  - 재로그인: **「김러너님 다시 오셨네요! 맞습니까?」**

**재로그인 확인 후**

- `localStorage.chunbaekSessionToken` 저장 (응답 token)
- 토스트: "다시 오신 걸 환영해요" → **오늘** 화면

### 16.4 ③ 프로필 생성

```
┌─────────────────────────────┐
│  김러너님의 프로필           │
├─────────────────────────────┤
│  닉네임                      │
│  김러너            (고정)    │
├─────────────────────────────┤
│  풀 코스 목표 기록 *         │
│  [ 4 ] 시간 [ 30 ] 분        │
│  춘천마라톤(또는 시즌 대회)   │
│  목표 완주 시간              │
├─────────────────────────────┤
│  기존 풀 PB (선택)           │
│  [ 4 ] 시간 [ 52 ] 분        │
│  없으면 비워두세요           │
├─────────────────────────────┤
│  [    프로필 만들기    ]     │
└─────────────────────────────┘
```

**필드**

| 필드 | 필수 | 형식 | 비고 |
|------|------|------|------|
| `nickname` | — | 표시만 | ②에서 선택, 수정 불가 |
| `goalMarathonNetTime` | **필수** | 시·분 (net) | 출정식 개인 목표와 연결 |
| `existingPbNetTime` | 선택 | 시·분 (net) | 없으면 `null` |

**검증**

- 목표 기록: 2:00:00 ~ 7:00:00 범위 (합리적 마라톤)
- PB 입력 시 목표보다 느려도 허용 (도전 목표)

**저장 (`POST create-profile`, Bearer 없음 — 최초 1회)**

- body: `{ memberId, goalMarathonNetTime, existingPbNetTime? }` — 서버가 `participant`·미가입 여부 검증
- `chunbaekS3.profileComplete: true`, 목표·PB 저장 + **session token 발급**
- 동시 가입 방지: 트랜잭션으로 `profileComplete` 재확인
- 성공 → ④로 이동 + `localStorage.chunbaekSessionToken` 저장

### 16.6 재진입·예외 처리

| 상황 | 동작 |
|------|------|
| localStorage token 유지 | 온보딩 전부 생략 → **오늘** |
| 새 기기, 이미 가입함 | ②에서 본인 선택 → `link-device` → ③④ 건너뜀 → **오늘** |
| ③ 중 이탈 (미완) | token 없거나 `profileComplete false` → **③부터** 재개 |
| 잘못된 닉네임 (신규) | 운영진 admin에서 `chunbaekS3` 프로필 초기화 후 재가입 |
| 「다른 사람으로」 | token 삭제 → ①부터 (이미 `profileComplete`면 ③④ 생략) |
| 운영진 | admin: `profileComplete` 리셋, 목표·PB 수정, session revoke |

> **보안 (C1):** 명단 선택 UX는 DMC와 같지만, **API는 session token으로 신원 검증**한다. 링크만으로 타인 출석·조회 불가. 대리 선택 위험은 운영·단톡 폐쇄로 완화.

### 16.7 API (`/api/chunbaek`)

베이스: §10.2. 회원 action은 **token 필수** (`Authorization: Bearer` 또는 `?token=`). Admin action은 `verify-admin` 후.

| action | 메서드 | 인증 | 용도 |
|--------|--------|------|------|
| `members-roster` | GET | 없음 | 참가자 명단 (`participant`만, `nickname`, `profileComplete`) |
| `create-profile` | POST | 없음 (1회) | 신규. `{ memberId, goalMarathonNetTime, existingPbNetTime?, resolutionText? }` → token 발급 |
| `link-device` | POST | 없음 (1회) | 재로그인. `{ memberId }` → `profileComplete` 확인 후 token 발급 |
| `my-profile` | GET | token | 본인 프로필·출석 요약 |
| `today-slot` | GET | token | 오늘 슬롯 + 본인 출석 상태 |
| `save-attendance` | POST | token | `{ slotId, attended, note?, photoUrl? }` |
| `my-timeline` | GET | token | 내 100일 타임라인 (주차별) |
| `team-summary` | GET | token | 팀 집계·참가자 목표 (PB 제외) |
| `verify-admin` | POST | 비밀번호 | 운영진 세션 (DMC 패턴) |
| `admin-grid` | GET | admin | 주차별 출석 그리드 |
| `admin-set-attendance` | POST | admin | 출석·예외 수정 |
| `admin-import-slots` | POST | admin | 100슬롯 CSV import |

| 필드 | 팀 화면 | 나 탭 |
|------|---------|-------|
| `goalMarathonNetTime` | **공개** | 표시·수정 |
| `existingPbNetTime` | **비공개** | 표시·수정 |
